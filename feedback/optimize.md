# 프롬프트 최적화 가이드

문제 풀이 후 메인 MD 및 2티어 MD의 개선 사항을 정리한다.
사용자가 "최적화" 또는 "프롬프트 개선"이라고 하면 아래 형식으로 출력한다.

============================================================
## 최적화 분석 형식
============================================================

```
[최적화 제안]

대상 파일:
(메인 MD / categories/web.md / categories/pwn.md 등 2티어 MD)

문제점:
(이번 풀이에서 프롬프트가 비효율적이었거나 부족했던 부분)

원인:
(왜 그 부분이 문제였는지)

제안:
(추가/수정/삭제할 내용 — 구체적인 문장 단위로 제시)

기대 효과:
(개선 후 어떤 분석력 향상이 기대되는지)
```

============================================================
## 최적화 판단 기준
============================================================

아래 상황이 발생했으면 해당 파일의 최적화가 필요하다:

**추가가 필요한 경우**
- 이번 문제에서 반복적으로 확인한 패턴이 프롬프트에 없었던 경우
- 특정 공격 기법이 프롬프트에 없어서 떠올리는 데 시간이 걸린 경우
- 새로운 카테고리 또는 서브카테고리가 필요하다고 느낀 경우

**수정이 필요한 경우**
- 프롬프트의 특정 규칙이 이번 문제에서 오히려 방해가 됐던 경우
- 순서나 우선순위가 잘못 설정되어 비효율적인 분석 흐름이 생긴 경우
- 설명이 모호해서 잘못된 방향으로 분석한 경우

**삭제가 필요한 경우**
- 실제 풀이에서 한 번도 활용되지 않는 내용이 누적되는 경우
- 다른 규칙과 중복되어 컨텍스트만 낭비하는 경우

============================================================
## 최적화 적용 원칙
============================================================

- 한 번의 풀이 경험으로 즉시 삭제하지 않는다 — 2~3회 불필요함이 확인된 후 삭제
- 추가는 즉시, 삭제는 신중하게
- 수정/삭제 시 이유를 설명하고 사용자 confirm 후 적용한다
- git commit은 세션 마무리 시 한 번, push는 사용자 요청 시에만 수행한다

끝.
